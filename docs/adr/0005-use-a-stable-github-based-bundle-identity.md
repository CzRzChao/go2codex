# Use a stable GitHub-based bundle identity

Go2Codex will use `io.github.czrzchao.go2codex` as the Settings App bundle identifier from the Personal MVP onward, aligned with the intended `CzRzChao` GitHub namespace rather than an unowned domain. The embedded Toolbar Launcher will use `io.github.czrzchao.go2codex.launcher`. Keeping both identifiers stable preserves the intended preference domains and Launch Services registrations, but identifiers alone do not guarantee Automation authorization continuity for ad-hoc-signed Personal rebuilds. Both entry points display as `Go2Codex`; `Launcher` remains an internal executable and bundle distinction rather than user-facing branding.

Debug builds will instead use `io.github.czrzchao.go2codex.debug` and `io.github.czrzchao.go2codex.debug.launcher`, with both entry points displayed as `Go2Codex Debug`. Their outer product is `Go2CodexDebug.app` with executable and `CFBundleName` `Go2CodexDebug`; Personal Release remains `Go2Codex.app` with `Go2Codex`. This filesystem-name separation is required in addition to distinct bundle identifiers: a live validation showed that Launch Services name-based lookup could select a Debug wrapper still named `Go2Codex.app`, even though its identifier and plist name were Debug-specific. Bundle verification therefore rejects a configuration whose outer wrapper or executable has the other configuration's name. For a nested Launcher invocation, macOS attributes TCC responsibility to the corresponding outer bundle identity. Both outer and nested bundles declare the Apple Events entitlement and localized usage description, while Apple Event sending code remains Launcher-only and ordinary Settings launches send none. Separate preference domains and outer identities keep stored settings and TCC responsibility configuration-specific, but either ad-hoc build may require consent again after rebuilding. Public Release uses Developer ID Application signing to establish its stable signed identity.
